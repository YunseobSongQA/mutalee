## 비주얼 아이덴티티 (앱 전체 무드)

앱 마스코트 "무탈이"의 분위기에 맞춰서 화면 전체의 무드와 색을 통일한다.
(무탈이: 둥글고 담백한 오프화이트 캐릭터, 머리 위 세이지 그린 새싹,
크림 스카프, 연핑크 볼터치. 미니멀하고 따뜻하고 무해한 느낌.)

### 색상 팔레트
- 베이스/배경: 오프화이트 #FAF9F6
- 텍스트/라인: 소프트 다크그레이 #3a3a3a
- 포인트(주요 액션·강조): 세이지 그린 #A9C29A
- 서브: 크림베이지 #E8E0D0
- 강조(포인트 감정): 연핑크 #F4C7C3
> 정확한 hex는 나중에 조정. 이 톤(저채도·따뜻함)만 지켜라.

### 무드 원칙
- 미니멀 + 여백 넉넉하게. 정보 빽빽하게 채우지 말 것
- 모든 카드/버튼 border-radius 크게, 각진 요소 지양
- 강한 그림자·화려한 그라데이션 금지. 아주 옅은 소프트 섀도우만
- 둥근 고딕 폰트(Pretendard 등), 편지 톤이라 자간·행간 여유 있게
- 전반적으로 담백하고 따뜻한 무해함 — 자극적이거나 쨍한 색 금지

### 지금 할 것
색·라운드·타이포를 CSS 변수(디자인 토큰)로 먼저 정의하고,
그걸 기준으로 화면 구현. 설계부터 보여주고 진행.

## 1. 이게 뭐냐 (목적)
사람은 잊는다. 좋은 것도, 소중한 것도, 지켜야 할 규칙도 잊는다.
차갑게 "일정 알림" 때리는 시스템 말고, **편안하고 기분 좋게 나를 도와주는 알림 시스템**을 만든다.
아버지·어머니의 잔소리 같은, 그러나 **내가 직접 정해둔 나를 위한 규칙**. 똑똑한 비서 같기도, 나를 안 무너지게 잡아주는 철학자 같기도 한 톤.

1차 목적은 나 자신을 위한 것. 장기적으로는 남들에게도 도움이 되는 무언가로 키운다.

## 2. 핵심 컨셉 — 카테고리별 노티
카테고리를 정하고, 각 카테고리 안에 "규칙(노티)"을 여러 개 둔다. 각 노티는 요일·시각·톤(페르소나)·문구를 가진다.

예시:
- **정신건강**: "필요 없는 알림들 정리하는 시간이에요" (특정 요일·시각)
- **주식**: "설정하신 주식 확인 시간입니다. 딱 지금만 보시고 나오세요" (그 시각에만 확인 유도)
- **인간관계**: "연인을 좋아했던 이유를 떠올려볼 시간이에요" (소중함을 잃지 않게)
- **아침 교양 한스푼**: 좋아하는 시집 한 구절 / 추천 클래식 한 곡 (매일 아침, 매일 다른 걸 결정론적으로 선택)

톤(페르소나)은 데이터로 고를 수 있게: `비서` / `철학자` / `부모잔소리` / `교양`.
문구 예: "○○님이 설정하신 주식 확인 시간 전달드립니다."

## 3. 기술 현실 (반드시 지킬 것 — 과장 금지)
- 플랫폼: **iOS 홈 화면 PWA**. 프레임워크 없이 브라우저에서 바로 도는 웹앱.
- iOS PWA는 앱이 **닫혀 있을 때 예약 알림을 자체적으로 못 쏜다.** `setTimeout` 스케줄은 앱 닫히면 죽는다.
- 진짜 백그라운드 예약 푸시는 웹푸시(서비스워커 + VAPID 구독 + **푸시를 쏴줄 서버**)가 있어야만 된다. → 이건 Phase 2.
- 그러니 **없는 기능을 있는 척 만들지 마라.** Phase 1은 "앱 열면 오늘 울릴 것들이 보이고, 앱이 켜진 동안 알림이 뜬다" 수준에서 **정직하게** 완결한다.

## 4. 페이즈 계획 (13번 규칙: 많으면 나눠서)
### Phase 1 — 서버 없는 로컬 PWA (지금 이것만 구현)
- 카테고리 + 노티 규칙을 데이터(JSON)로 관리.
- 앱 열면 **오늘/지금 시점에 해당하는 노티 카드**를 보여준다 (요일·시각 기준으로 필터).
- 노티 규칙 추가/수정/on-off 가능.
- 앱이 포그라운드로 켜져 있을 때 브라우저 `Notification`으로 알림 (권한 요청 포함). 닫혀 있을 때 안 되는 건 UI에 솔직히 안내.
- 매일 아침 교양: 시 구절/클래식 곡 카탈로그에서 **날짜 기반 결정론적 선택**(같은 날은 항상 같은 것).
- 홈 화면 추가용 manifest + 최소 서비스워커(오프라인 캐시 정도).
- 데이터 저장은 localStorage로 시작.

### Phase 2 — 진짜 예약 푸시 (나중, 별도 승인 후)
- 내 BountyOps 패턴 재사용: **Cloudflare Pages Functions + Cloudflare KV + 웹푸시(VAPID)**.
- KV에 노티 규칙 저장 → cron으로 시각 도래 시 푸시 발송.
- Phase 1의 순수 로직/데이터 스키마를 그대로 재사용할 수 있게 Phase 1에서 경계를 잘 그어둘 것.

---

## 5. 데이터 모델 (모듈 계약 — 이 형태 고정)
노티 규칙 하나의 형태:
```
{
  id: string,
  category: string,        // "mental" | "stock" | "relationship" | "culture" ...
  persona: string,         // "secretary" | "philosopher" | "parent" | "culture"
  title: string,
  message: string,         // "{name}님이 설정하신 ..." 처럼 치환 토큰 사용
  schedule: { days: number[], time: "HH:MM" },   // days: 0(일)~6(토)
  enabled: boolean
}
```
카테고리·페르소나·교양 카탈로그·기본 문구는 전부 **JSON 데이터로 분리**한다 (하드코딩 금지).

## 6. 순수 함수 (DOM·네트워크 없이 입출력만)
최소한 아래는 순수 함수로 격리하고, UI/저장소와 섞지 말 것:
- `getDueReminders(rules, now) -> reminder[]`  // 지금 시점에 해당하는 노티만 골라냄
- `renderMessage(reminder, profile) -> string`  // 문구 토큰 치환 ({name} 등)
- `pickDaily(catalog, dateSeed) -> item`         // 날짜 기반 결정론적 1개 선택
이 함수들은 UI를 갈아엎어도 안 건드려지게, Phase 2에서도 그대로 재사용되게 설계.

---

## 7. 파일/스택 제약
- **Vanilla JS only. 프레임워크·빌드 스텝·외부 의존성 0.** 브라우저에서 바로 실행.
- 폴더는 타입 말고 **기능별**로 (예: `reminders/` `culture/` `notify/` `data/`).
- 파일 수 최소화. 굳이 안 쪼개도 될 걸 미리 쪼개지 말 것.

## 8. 작업 방식 (이 순서 지켜)
1. **먼저 설계만**: 파일 구조 + 로직 흐름 + 위 데이터 스키마를 어떻게 쓸지 제안. 구현하지 말 것.
2. 내가 OK 하면 **Phase 1만** 구현 시작.
3. 막혀도 **중간에 질문하지 말고** 합리적으로 가정해서 끝까지 간 뒤, **가정을 마지막에 정리**해서 보여줘.
4. 수정할 땐 **바뀐 부분만** 출력. 안 건드린 코드/파일 통째로 다시 뿌리지 마.

## 9. 바이브코딩 규칙 (전부 준수)
1. 모든 대화는 한글로 소통할 것.
2. 변하는 이유가 다른 건 분리해라 — UI(자주 바뀜)/로직(가끔)/데이터(거의 안 바뀜)를 한 함수·한 파일에 섞지 마. 유지보수 설계의 8할.
3. 핵심 로직을 순수 함수로 격리 — classify(rows) → result처럼 DOM·네트워크 없이 입출력만. 테스트 되고, UI 갈아엎어도 안 건드리고, "이 함수만 고쳐"로 토큰 폭발 차단.
4. 자주 바뀌는 규칙·설정은 데이터로 빼라 — 임계값·규칙·문구를 하드코딩하지 말고 KV/JSON으로. 규칙 추가하는데 코드를 안 건드리게. (BountyOps KV 분리가 이거)
5. 폴더는 타입 말고 기능별로 — utils.js/api.js로 흩어놓지 말고 capture/ rules/ dashboard/로 묶어. "이 기능 관련 건 다 여기 있다"가 되게.
6. 모듈 경계(계약)를 고정 — 주고받는 데이터 형태를 정해놓고 지켜({type, score, reason}). 약속만 안 깨면 내부는 마음대로 바꿔도 영향 0 → 부분 수정 가능.
7. 코드 말고 설계부터 시켜라 — "구현하지 말고 파일 구조랑 로직 흐름만 먼저. OK하면 그때 시작."
8. 질문 금지 + 가정 명시 — "막혀도 묻지 말고 합리적으로 가정해서 끝까지. 가정은 끝에 정리."
9. 파일 수·스택 미리 못 박기 — "Vanilla JS only, 빌드 없이 브라우저에서 바로."
10. 수정은 전체 재작성 말고 변경분만 — "기존 파일 통째로 다시 쓰지 말고 바뀐 부분만, 안 건드린 코드는 출력하지 마."
11. 조기 추상화 금지 — 두 번까진 복붙, 세 번째 나올 때 추출.
12. 클로드 공동 작업자 표시 금지 — git commit 시 Co-Authored-By, Claude, Anthropic, "Generated with Claude" 같은 문구나 trailer를 커밋 메시지에 절대 포함하지 말 것.
13. 내용이 많으면 페이즈로 분리하여 조금씩 구현.
14. 개발자나 코드 읽기 약한 사람도 읽기 쉬운 구조로, 복잡하지 않게 최대한 간략하게 작성.
